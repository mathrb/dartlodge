# Internationalisation (i18n) — Design

**Date:** 2026-06-16
**Status:** Validated (design phase) — issues not yet created
**Epic scope:** Fully internationalize DartLodge into 7 languages, foundations first then per-feature migration.

---

## 1. Goal & scope

Make the entire DartLodge UI translatable and ship it in 7 languages:

**EN** (source/fallback), **FR**, **ES**, **DE**, **PT**, **IT**, **NL**.

The app is currently 100% hardcoded English (~51 presentation files, 200+ string literals). There is no existing localization setup; `intl ^0.20.2` is already a dependency. Settings already use a clean `@Riverpod(keepAlive: true)` + `SharedPreferences` pattern (`ThemeMode`) that the locale setting mirrors exactly.

This is **one epic** spanning foundations + per-feature migration. The epic closes only when the whole app is translated.

---

## 2. Decisions (locked)

| Decision | Choice | Rationale |
|---|---|---|
| Languages | EN, FR, ES, DE, PT, IT, NL | EN = source & fallback |
| Tooling | **gen_l10n** (official `flutter_localizations` + `intl` + ARB) | Zero new third-party package (`intl` already present, `flutter_localizations` is first-party); official backing/longevity; compile-time placeholder safety. Clean architecture confines strings to `presentation/`, so the `BuildContext`-based access is never a constraint — the main argument against the official solution does not apply here. |
| Generated code | **Not committed** (initially) | Generated into a `.gitignore`-d dir; `flutter pub get` regenerates it. Fallback if CI is fragile: commit the dir. |
| Locale UX | **Follow system + manual override** | Mirrors the existing `ThemeMode` (`system`/`light`/`dark`) pattern; testable; good first impression for non-English users. |
| Locale persistence | `SharedPreferences`, key `app_locale` | Same pattern as `ThemeMode`. A local preference is NOT a `GameEvent` (it does not change game state) → out of the event-stream flow. |
| Translation sourcing | AI-assisted (Claude), reviewed by the maintainer | Fast, free, consistent. gen_l10n fallback to EN is the automatic safety net for any incomplete ARB. |
| Migration unit | One feature per sub-issue; extraction + 7-language translation in the **same** PR | Avoids a second pass over the same files. |

### Tooling alternatives considered

Spectrum reviewed (2025-2026): official `gen_l10n`, `slang`, `easy_localization`, GetX. For production apps the two serious candidates are `gen_l10n` and `slang`. `slang` offers `context`-free type-safe access (`t.x`) and integrates with the existing `build_runner` workflow, but adds a third-party dependency. Rejected in favour of `gen_l10n` because the only real `slang` advantage (no `BuildContext`) is moot under clean architecture, and the "no new dependency + official + long-term stable" criterion wins for a disciplined open-source side project.

> ⚠️ **Common misconception:** comparison tables mark the official solution "runtime switching: No". This is misleading — the official solution **does** switch language at runtime by varying the `locale` passed to `MaterialApp` (rebuilds the tree). That is exactly what `LocaleSetting` does. Our override UX works.

## 3. Architecture

### 3.1 Tooling config

`l10n.yaml` (repo root, next to `pubspec.yaml`):
```yaml
arb-dir: lib/l10n/arb
template-arb-file: app_en.arb
output-localization-file: app_localizations.dart
output-dir: lib/l10n/gen
synthetic-package: false      # synthetic `package:flutter_gen` is deprecated/removed
nullable-getter: false
```

`pubspec.yaml`:
- add `flutter_localizations: { sdk: flutter }`
- add `flutter: generate: true`
- `intl ^0.20.2` already present — unchanged

> The synthetic `package:flutter_gen` is deprecated; modern gen_l10n emits source into `output-dir`, imported directly. `generate: true` is required.
> Refs: [Flutter breaking change — l10n source generation](https://docs.flutter.dev/release/breaking-changes/flutter-generate-i10n-source).

### 3.2 File layout

```
lib/l10n/
  arb/
    app_en.arb          # template (source + fallback)
    app_fr.arb  app_es.arb  app_de.arb
    app_pt.arb  app_it.arb  app_nl.arb
  gen/                  # GENERATED, .gitignore-d
    app_localizations.dart
    app_localizations_en.dart … (one per language)
  supported_locales.dart   # single source of truth for the language list
```

`.gitignore`: add `lib/l10n/gen/`.

`supported_locales.dart` — reused by both `MaterialApp.supportedLocales` and the selector:
```dart
const kSupportedLocales = [
  Locale('en'), Locale('fr'), Locale('es'),
  Locale('de'), Locale('pt'), Locale('it'), Locale('nl'),
];
```

### 3.3 Locale provider & persistence

`lib/features/settings/presentation/providers/locale_provider.dart` — mirrors the `Settings`/`ThemeMode` pattern:
```dart
@Riverpod(keepAlive: true)
class LocaleSetting extends _$LocaleSetting {
  static const _kKey = 'app_locale';   // language code, or absent = system

  @override
  Future<Locale?> build() async {
    final prefs = await ref.watch(sharedPreferencesProvider.future);
    final code = prefs.getString(_kKey);
    return code == null ? null : Locale(code);
  }

  Future<void> setLocale(Locale? locale) async {
    final prefs = await ref.read(sharedPreferencesProvider.future);
    if (locale == null) {
      await prefs.remove(_kKey);          // back to "System"
    } else {
      await prefs.setString(_kKey, locale.languageCode);
    }
    state = AsyncData(locale);
  }
}
```
Generated provider: `localeSettingProvider` (only the `Notifier` suffix is stripped; the `Setting` suffix stays — verify the `.g.dart`).

Value semantics:
- `null` ⇒ **System**: resolved via `localeResolutionCallback` (device language if supported, else EN).
- `Locale('fr')` etc. ⇒ explicit override.

No drift table, no `GameEvent` — a local preference does not change game state.

### 3.4 MaterialApp wiring

`lib/app/app.dart` (`MaterialApp.router`, already a `Consumer` for `themeMode`):
```dart
final locale = ref.watch(localeSettingProvider).value; // Locale? — null = system

return MaterialApp.router(
  // …existing theme/router…
  locale: locale,
  supportedLocales: kSupportedLocales,
  localizationsDelegates: AppLocalizations.localizationsDelegates,
  localeResolutionCallback: (deviceLocale, supported) {
    if (locale != null) return locale;                  // explicit override
    final match = supported.firstWhereOrNull(
      (l) => l.languageCode == deviceLocale?.languageCode,
    );
    return match ?? const Locale('en');                 // robust fallback
  },
);
```
- `AppLocalizations.localizationsDelegates` already bundles the Material/Widgets/Cupertino delegates + ours.
- Changing `locale` via the notifier rebuilds `MaterialApp` → whole tree re-renders in the new language (runtime switching).
- `firstWhereOrNull` from `package:collection` (already transitively present), else `try/catch` on `firstWhere`.

### 3.5 Foundation strings & language selector

Foundation ARB carries only the selector's own strings (everything else stays hardcoded EN until per-feature migration):

`lib/l10n/arb/app_en.arb`:
```json
{
  "@@locale": "en",
  "settingsLanguageLabel": "Language",
  "@settingsLanguageLabel": { "description": "Settings row label for language selection" },
  "languageSystemDefault": "System default",
  "@languageSystemDefault": { "description": "Option to follow the device language" }
}
```

**Language names are NOT ARB keys** — displayed as autonyms (each language names itself) via a constant map: English / Français / Español / Deutsch / Português / Italiano / Nederlands. Standard UX, and avoids 7×7 translations of language names.

Selector widget `_LanguageSelector` in the Settings page, modeled on the existing `_ThemeModeSelector`: a row/`ListTile` opening a list of "System" + the 7 autonyms, checked on the current value; `onChanged` → `localeSettingProvider.notifier.setLocale(...)` (`null` for System). The "Language" row label itself is localized, so selecting FR immediately shows "Langue" / "Système" — the visible end-to-end demo.

## 4. Testing

- **Selector widget test** — override `localeSettingProvider` + `sharedPreferencesProvider`; tap "Français" → asserts `setLocale(Locale('fr'))` and persistence.
- **End-to-end widget test** — pump app with `localeSettingProvider` = `Locale('fr')`; assert Settings shows "Langue"/"Système"; with `null` + EN device → "Language".
- **Fallback test** — `localeResolutionCallback` with unsupported device locale (`ja`) → `en`.
- **ARB integrity test** (key guard) — load all 7 ARB files, assert identical key set to `app_en.arb`. Prevents a missing key in any language (analogous to `rules_registry_test`). This test grows with every migration sub-issue.

## 5. CI

- Confirm `flutter pub get` regenerates `lib/l10n/gen/` before `analyze`/`test`. **Risk** the maintainer flagged: if a runner runs `analyze`/`test` before generation, the import breaks. Mitigation: commit `lib/l10n/gen/` (drop from `.gitignore`).
- `flutter analyze --no-fatal-infos` must stay green.

## 6. Sub-issue breakdown

One epic. One PR per sub-issue, shipped serially (review via `code-review` → green CI → merge). Each sub-issue describes symptom/objective + an "Avant implémentation" (`/plan`) section — not the coded solution. Attach as GitHub native sub-issues (per epic #230/#477 structure).

**Foundations**
- **SI-1** — gen_l10n setup: `l10n.yaml`, `pubspec` (`flutter_localizations` + `generate: true`), 7 skeleton ARB (2 keys), `.gitignore`, `kSupportedLocales`.
- **SI-2** — `LocaleSetting` provider + `SharedPreferences` persistence + tests.
- **SI-3** — `MaterialApp` wiring (delegates, `supportedLocales`, `localeResolutionCallback`) + fallback test.
- **SI-4** — Language selector in Settings + localized "Language" row + widget tests.

**Migration + translation (one feature per sub-issue; ARB extraction + 7 languages in the same PR)**
- **SI-5** — `settings/` (beyond the 2 foundation keys)
- **SI-6** — `players/`
- **SI-7** — `history/`
- **SI-8** — `statistics/`
- **SI-9** — `game/` (largest — X01/cricket/practice boards; may need re-splitting)
- **SI-10** — `auto_scorer/`

The app stays functional at every step: unmigrated features remain hardcoded EN.

## 7. Constraints honored

- No new third-party dependency (`intl` already present; `flutter_localizations` first-party).
- Strings consumed only in `presentation/` — `domain/` stays Flutter-free.
- No `GameEvent` / drift for a UI preference.
- Branch-per-change, serial PRs, `code-review` skill before merge.
