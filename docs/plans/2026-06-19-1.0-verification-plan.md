# Plan de vérification end-to-end — pré-1.0

> **Statut :** document parapluie (high-level). Date : 2026-06-19.
> Décrit *quoi* vérifier sur 5 axes avant la 1.0. Le *comment* détaillé (cas, étapes,
> attendus) vivra dans un document par axe (voir §9).

---

## 1. Objet, périmètre & matrice de synthèse

### Objet
Cadrer un cycle de vérification end-to-end avant la 1.0, pour **découvrir les bugs et
incohérences UX/design résiduels** et **produire un filet de régression réutilisable**.

### Approche hybride
Chaque axe combine :
1. une **charte exploratoire** par surface — missions + heuristiques pour débusquer
   l'inconnu (jetable) ;
2. un **socle de scénarios critiques scriptés** — candidats à devenir des specs
   `@playwright/test` dans `e2e/` (durable).

### Deux rails
- **Rail web** (Playwright + `window.dartlodgeSim`) — couvre le scoring de tous les jeux,
  l'UX/flux, le design visuel par screenshot, et — **via émulation de flèches lancées** —
  le flux **assist auto-scoring (logique + correction + UX)**.
- **Rail device** (manuel sur téléphone) — réservé à ce que le web ne peut pas atteindre :
  vraie caméra / `CameraPreview` / précision modèle, sons (audioplayers), haptique, crash
  release Android (R8/rc144), runtime sqlite3 minSdk23, orientation/mount, layout réel à
  distance.

Le **rail est un attribut de chaque item** (tagué `[web]` / `[device]`), pas une catégorie
structurante — d'où l'organisation du plan par axe.

### Hors périmètre
- Précision/recall du modèle YOLO et boucle de retrain → **#393, repo `deep-darts-probe`**.
- i18n traité **en dernière vague** (axe 5), une fois l'UI stabilisée.

### Matrice de synthèse (axe × rail)

| Axe | Web (auto) | Device (manuel) |
|---|---|---|
| Correctness scoring | ●●● (sim) | ○ (sanity) |
| UX / flux | ●●● | ●● |
| Design visuel | ●●● (screenshots) | ●● (#477 distance) |
| Données / persistance | ●●● | ○ (minSdk23) |
| i18n *(dernière vague)* | ●●● | ● |

`●` = couverture forte · `●●` = partielle · `○` = marginale.

## 3. Axe 1 — Correctness (scoring)

**Question centrale :** le score affiché et l'état du jeu sont-ils toujours justes ?
Les moteurs sont déjà couverts par les tests unitaires/contract ; ici on cible
l'**intégration bout-en-bout** (saisie → event → projection → UI) et les **cas-limites de fin**.

**Rail :** quasi tout `[web]` via sim bridge ; un `[device]` sanity en fin de cycle.

**Heuristiques de chasse :**
- **X01** — bust (retour score, tour avancé), checkout double/master/straight, double-in
  non-entré (tour = 0), highest checkout, buckets +100/140/180, cap de rounds par leg →
  décision à la standing.
- **Cricket** — overflow `max(0, hits+mult-3)`, Bull = 25/overflow, fermeture/lock de cible,
  cut-throat (points adverses), no-score, **marks = cibles actives seulement** (régression
  #569), target modes random/crazy (rotation, `CrazyTargetsRolled`).
- **ATC / Shanghai / Catch 40 / Bob's 27 / Checkout / Count Up** — progression de cible,
  conditions succès/échec, busted (Bob's 27), fin de drill, attempts/successes.
- **Transverse** — undo (y compris franchissement de frontière de tour), correction
  (`DartCorrected`, `original_event_id`), replay déterministe après reload.

**Scénarios scriptés critiques :**
1. X01 501 double-out, 2 joueurs → checkout exact → post-game cohérent.
2. Cricket 3 joueurs jusqu'au DB gagnant (`cricket_3players.spec.ts` existe — auditer/étendre).
3. Bust X01 + undo → retour d'état exact.
4. Cricket #569 : segment non-cible affiché mais **non compté** en marks.

**Oracle :** `docs/games/*.transitions.md` · `docs/statistics/*.projections.md`.

## 4. Axe 2 — UX / flux

**Question centrale :** l'utilisateur peut-il traverser chaque parcours sans rester coincé,
perdu, ou face à un comportement surprenant ? On regarde les *transitions* et les *états*.

**Rail :** majorité `[web]` ; `[device]` pour back physique, haptique, ressenti assist caméra.

**Heuristiques de chasse :**
- **Navigation & back** — `go()` vs `push()` : Home → Stats/History/Players/Settings
  poppable ; le back ne quitte jamais l'app par surprise ni ne saute une modal non
  acquittée ; variant selection sous `PopScope`.
- **États non-heureux** — listes vides, loading (`AsyncValue`), erreurs (3 états gérés
  partout, pas de `.value!` nu).
- **Modals & interruptions** — bust / leg / win / busted / toast succès : bon timing,
  acquittement obligatoire, pas de double-déclenchement, pas de « blow past » via auto-advance.
- **Undo / correction** — dispo au bon moment, désactivé quand vide ; correction band → sheet.
- **Setup** — joueur auto-sélectionné, `canStart` cohérent, config invalide bloquée, variant flows.
- **Auto-scoring assist** — émulation de flèches : confirm-before-clear, auto-advance opt-in
  guardé par `_sawDartsThisTurn`, no-op si modal/jeu fini.
- **Jeu en cours** — reprise après reload (replay), sortie menu → home (pas post-game),
  partie terminée = read-only.

**Scénarios scriptés critiques :**
1. Parcours nominal Home → setup → partie → post-game → home, back testé à chaque étape.
2. Reload en pleine partie → état restauré.
3. Auto-advance opt-in : board clear émulé → tour avance sans sauter de joueur.
4. États vides des 4 écrans principaux.

**Oracle :** `docs/UI_SCREEN_FLOWS_V3_FINAL.md` + règles back de `CLAUDE.md`.

## 5. Axe 3 — Design visuel

**Question centrale :** chaque écran respecte-t-il le système de design, et l'ensemble est-il
cohérent ? **Méthode visuelle** : screenshot puis inspection œil + comparaison aux tokens.

**Rail :** `[web]` pour conformité tokens et cohérence inter-écrans (viewport fixe) ;
`[device]` pour la lisibilité réelle à distance (#477).

**Heuristiques de chasse :**
- **Tokens couleur** — aucune couleur en dur ; score actif `onSurface`, inactif/adversaire
  `onSurfaceVariant`, cible practice `primary` ; over-line `label-sm` héro = `primaryFixed`.
- **Typo** — noms joueurs board `labelMedium` ALL-CAPS `letterSpacing 1.2` ; nombres via
  `StatFormatter` (jamais `toStringAsFixed` inline) ; score jamais en `FittedBox(scaleDown)`.
- **Espacements & layout** — paddings/gaps conformes `DESIGN_SYSTEM.md` ; alignement strips
  (X01/Cricket/Practice), bandes proéminentes.
- **Camera-first** — bande 110px `scoreMedium`, `HeroMetricWidget`, FittedBox sur labels,
  vignette, `CricketMarkPainter` partagé (#477).
- **Cohérence inter-écrans** — mêmes composants stylés pareil (cartes history, badges AVG,
  onglets stats) ; thème clair/sombre.
- **Heatmap** — colormap/KDE lisibles, board correct, légende.
- **États transitoires** — pas de flash, skeletons cohérents.

**Scénarios scriptés (visuels) :**
1. **Galerie de screenshots** : chaque surface §2 capturée à un viewport de référence → revue.
2. *(post-1.0, optionnel)* baseline screenshots pour diff de régression visuelle.

**Oracle :** `docs/design/DESIGN_SYSTEM.md` + epic #477.

## 6. Axe 4 — Données / persistance

**Question centrale :** ce qui est calculé et stocké est-il exact et durable ? Les
statistiques sont des **projections rejouées** (jamais stockées) — on vérifie que le replay
produit les bons chiffres et que history/succès/heatmap reflètent fidèlement les parties.

**Rail :** quasi tout `[web]` (drift/IndexedDB ≈ natif) ; `[device]` pour le **runtime
sqlite3 minSdk23** (non vérifié).

**Heuristiques de chasse :**
- **Stats = projections** — moyennes X01, checkout %, highest checkout, buckets, MPR/MPT
  cricket, first-nine (présents seulement si `TurnStarted` émis), per-game/career/per-leg cohérents.
- **Scope des resets** — Turn sur `TurnStarted`, Leg sur `LegCompleted`, Match sur
  `GameCompleted` ; aucun autre.
- **Mark-bucket cricket** — exact-N (`getGameStats`/leg) vs ≥-N (`getPlayerStats`) selon l'écran.
- **GameStats.gameType** — toujours renseigné (y compris early-return darts vides) → labels
  MPR vs PPR justes.
- **Historique** — liste paginée, detail = leg breakdown juste, filtres, parties complétées read-only.
- **Achievements** — déblocage au bon moment (NineDarter, Games501), persistés, pas de
  re-déblocage, toast unique.
- **Heatmap** — positions x/y persistées (DartThrown payload + `dart_throws.x/y`),
  manual/corrected → null, agrégation all-time par type filtrée `isComplete`.
- **Persistance** — reload → données intactes ; multi-parties (tri `(game_id, local_sequence)`).

**Scénarios scriptés critiques :**
1. Partie connue → stats → chaque chiffre = valeur calculée à la main.
2. 9-darter émulé → succès débloqué + toast.
3. Reload → history + stats persistent.
4. Heatmap : N fléchettes à positions connues → densité au bon endroit.

**Oracle :** `docs/statistics/*` · `docs/DATABASE_DDL.md` · epic #571.

## 7. Axe 5 — i18n *(dernière vague)*

**Question centrale :** l'appli est-elle correcte et soignée dans les 7 langues ? Traité en
dernier : passe transverse qui ne vaut le coup qu'une fois l'UI stabilisée par les axes 1–4.

**Rail :** `[web]` pour l'essentiel (changement de langue + screenshots des 7 locales) ;
`[device]` marginal (polices non-latines).

**Heuristiques de chasse :**
- **Clés manquantes / fallback** — aucune clé brute affichée, aucun fallback silencieux EN.
- **Débordement** — langues à mots longs (DE/FI) ne cassent pas boutons/bandes/badges ; pas
  de troncature laide ; FittedBox-sur-labels (#477) absorbe.
- **Jargon darts** — termes consacrés par langue (checkout, ton, bust, leg, marks…) —
  web-search en cas de doute plutôt que deviner.
- **Pluriels & interpolation** — compteurs (« 1 leg » vs « 3 legs »), placeholders `{name}`.
- **Cohérence** — même concept = même terme partout.
- **Formats** — nombres/décimales via `StatFormatter`, cohérents selon locale.

**Scénarios scriptés critiques :**
1. Boucle sur les 7 locales → screenshot des écrans à fort texte (setup, post-game, stats,
   settings) → revue débordement/clés manquantes.
2. Parcours nominal complet en DE pour stresser les layouts.

**Oracle :** epic #505 (gen_l10n, ARB) + règle de traduction par recherche.

## 8. Convention de findings & workflow

**Carnet de findings (doc d'abord, issues ensuite).** Toutes les observations vont dans un
seul doc markdown vivant pendant le cycle ; les issues GitHub sont créées **par lot après
revue ensemble**, pour éviter doublons et bruit.

**Format d'un finding :**
```
### F-NNN — <titre court symptôme>
- Axe : Correctness | UX | Design | Données | i18n
- Surface : <jeu/écran de la §2>
- Rail : web | device
- Sévérité : P0 | P1 | P2
- Observé : <ce qui se passe>
- Attendu : <ce qui devrait se passer + oracle/réf>
- Preuve : <screenshot / étapes de repro / spec>
- Statut : à trier | confirmé | issue #N | won't-fix (raison)
```

**Sévérité (calquée sur le tracker) :**
- **P0** — sortie utilisateur fausse ou flux cœur cassé (mauvais score, crash, partie infinissable).
- **P1** — incohérence de correctness/architecture, ou UX/design dégradant nettement sans bloquer.
- **P2** — hygiène : polish design, micro-incohérence, dérive de doc.

**Règle anti-faux-positif.** Un finding est une *hypothèse*. Avant de le promouvoir en issue :
confirmer la repro + vérifier l'oracle. « Won't-fix + documenter le scope » est une issue
valide (ex. couleurs dartboard domaine, alpha literals component-local).

**De finding à issue.** Symptôme seul (pas de cause/fix), section « Avant implémentation »
forçant un `/plan`, sous-issues natives si epic, *aucune* issue « préparée » sans feu vert
explicite.

**De finding à régression.** Les findings P0/P1 reproductibles côté web deviennent des specs
`@playwright/test` dans `e2e/`.

## 9. Prochaines étapes & séquencement

### Hiérarchie documentaire
```
docs/plans/2026-06-19-1.0-verification-plan.md   ← ce doc (parapluie)
docs/testing/axis-1-correctness.md               ← détail (à venir)
docs/testing/axis-2-ux-flux.md
docs/testing/axis-3-design.md
docs/testing/axis-4-data-persistence.md
docs/testing/axis-5-i18n.md                      ← dernière vague
docs/testing/findings-2026-06.md                 ← carnet de findings vivant
```

### Ordre d'exécution
1. **Axes 1 + 2 ensemble** (Correctness + UX) — un seul passage par jeu, deux angles.
2. **Axe 3** (Design) — passe screenshot dédiée, écran par écran.
3. **Axe 4** (Données) — vérif sur parties à valeurs connues.
4. **Axe 5** (i18n) — en dernier, UI stabilisée.
5. **Rail device** — session manuelle consolidée à la fin, checklist dérivée des items `[device]`.

### Pré-requis technique
- App servie (`flutter run -d web-server`), sim bridge accessible.
- Audit rapide des specs `e2e/` existantes (valides vs scratch).

### Critère de fin de cycle
Chaque surface §2 couverte (ou écartée justifiée) sur les axes 1–4 ; findings triés ;
régressions P0/P1 web codifiées ; checklist device remise pour la passe manuelle.

### Définition de succès pour la 1.0
Zéro P0 ouvert · P1 triés (corrigés ou consciemment reportés) · P2 documentés.
